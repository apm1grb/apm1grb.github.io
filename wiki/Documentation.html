    Documentation

    Introduction

    The documentation for the BVMS UI Automation project is provided in a modern and state of the art model. It is based on these sources and perspectives:

    • XML Comments
      this is the in-source XML-based code documentation with triple slash (/// <summary>...</summary>).
    • Wiki
      which is hosted on Azure DevOps and based on MarkDown as the markup language and Git as the storage to interact with.
      It is located here: Wiki for BVMS UI Automation

    To keep the documentation parts as most as possible together we use the technique of Git sub-modules. Git sub-modules allow a Git ("main") repository to reference other ("sub") repositories. This gives the possibility to divide a project into several containers (e.g. because of different life cycle or usage needs) but bundle them together at a top level.

    In our concrete case we had to include two Git sub-modules

    • BVMS UI Automation project Wiki
      this Wiki is a Git repository itself containing the MarkDown wiki pages
    • DocFX based Online documentation
      GitHub Pages allow easily to host (static) web sites. These web sites have to be provided in a (dedicated) Git repository. So the output of DocFX (the static web sites) are committed to this Git repository and then pushed to the remote (github.io).

    Advice: To clone a Git repository together with its sub-modules use the option --recursive

    Folder structure

    The DevTree's doc-folder contains the project`s documentation itself and the configuration files for DocFX to process the content.
    The content and intention in detail:

    • folder doc itself

      • artifacts to configure the DocFX documentation system

        • docfx-json
          the DocFX project file
        • toc.yml
          the table of content file to hook up the wiki-pages
        • index.md
          the landing page for the DocFX generated online documentation

    • sub-folder wiki

      • this is the repository of the Git sub-module from BVMS UI Automation project Wiki

      • it was included via Git´s submodule sub-command

        git submodule add
  git@ssh.dev.azure.com:v3/<organization>/<project>/<repository>.wiki doc/wiki

    • sub-folder sdk

      • contains the metadata for the SDK part of the documentation
      • extracted from projects ConfigClient.DomainModel and ConfigClient.Sdk

    • sub-folder api

      • contains the metadata for the API part of the documentation
      • extracted from projects ConfigClient.Api and ConfigClient.ConsoleApp

    • sub-folder sdk-api

      • artifacts to configure the DocFX documentation system
      • merges content based on SDK and API metadata together

    • sub-folder _site

      • this is the repository of the Git sub-module with the static web-sites as an output of DocFX

      • the repository is connected to the web site hosting platform (e.g. github.io as a remote)

        git submodule add git@<alias>:<username>/<username>.github.io.git doc/_site

    DocFX

    To process the content of these sources we use DocFX which is the successor of SHFB (Sandcastle Help File Builder). DocFX is platform-agnostic and runs perfectly also under .NET Core. DocFX creates static web-sites which can easily hosted somewhere.

    To update the documentation these steps are essential:

    1. Update the working tree to the latest stand of the Wiki content
    2. Extract the XML metadata of the source code (the XML comment) with DocFX
    3. Process the documentation input with DocFX to create static web sites as output
    4. Commit the output of DocFX to the "online documentation" repository connected to hoster
    5. Push this "online documentation" repository

    Follow the next sections to do that:

    Update the working tree

    1. Open Git Bash in/with sub-folder .../doc/wiki/
    2. Run command git fetch to get the latest stand into your local repository
    3. Run command git status to ensure that nothing is uncommitted and the local and remote branch are at the latest stand
      • the output should be something like: Your branch is up to date with 'origin/wikiMaster'.
      • if not: stop further processing and first commit changes, rebase, ...
    4. Run command Git pull --ff-only to get the latest stand in your working tree

    Extract the XML metadata & process the documentation input

    For the following steps the DocFX toolset needs to be installed on your machine.
    Use Chocolatey with choco install docfx

    1. Open Command Prompt in/with sub-folder .../doc/
    2. Run command docfx metadata doc\docfx.json -o doc to extract the metadata of the source code
    3. Run command docfx build .\doc\docfx.json to create the static web-sites

    Publish the updated documentation

    To have the updated documentation online available the repository containing the static web-pages must be updated and pushed.

    1. Open Git Bash in/with sub-folder .../doc/_site_/
    2. Run command git commit -am "update DocFX documentation" to commit the changes
    3. Run command git push to update the remote

    Frequently used CLI commands

    Hint: The following commands are supposed to be used inside a Git Bash or Developer Prompt for Visual Studio. Therefore they use forward slashes (/) or backward slashes (\) for folder separation. So please use the right CLI environment to run the program.

    Recommendation: Use the Developer Prompt for Visual Studio for everything around compiling the codebase and the Git Bash for everything around versioning.

    Hint: All commands are treated as used from the project root folder (aka DevTree root folder).

    Compilation/Build

    msbuild .\src\Bvms.UiAutomation.sln /t:Restore
msbuild .\src\Bvms.UiAutomation.sln

    Run unit tests

    Bash:

    ./packages/xunit.runner.console/tools/net472/xunit.console.exe ./src/ConfigClient.DomainModel/ConfigClient.DomainModel.Tests/bin/Debug/ConfigClient.DomainModel.Tests.dll
./packages/xunit.runner.console/tools/net472/xunit.console.exe ./src/UseCaseTests/UseCaseTests/bin/Debug/Bvms.UiAutomation.UseCaseTests.dll
./deploy/xunit.console.exe ./deploy/Bvms.UiAutomation.UseCaseTests.dll

    Command Prompt:

    .\packages\xunit.runner.console\tools\net472\xunit.console.exe .\src\ConfigClient.DomainModel\ConfigClient.DomainModel.Tests\bin\Debug\Bvms.UiAutomation.ConfigClient.DomainModel.Tests.dll
.\packages\xunit.runner.console\tools\net472\xunit.console.exe .\src\UseCaseTests\UseCaseTests\bin\Debug\Bvms.UiAutomation.UseCaseTests.dll

    Git

    Hint: Some of the following Git commands make use of aliases. Ensure that your personal .gitconfig contains these aliases. You can read more about the default configuration in the Baseline chapter of the Installation Guideline .

    show the status

    git st

    switch to branch; Create new branch and switch to it

    git co <branch-name>
git co -b <new-branch-name>

    merge or rebase content of other branch in current branch

    Important: Use always the --ff-only option for a merge to avoid a 2-way merge!

    git merge <other-branch-name> --ff-only
git rebase <new-branch-name>

    synchronization between local and remote repository

    git fetch --all
git push
git pull --ff-only

    XUnit

    Categories, Trait

    To group unit test in a way which makes sense for a project, the Attribute Trait is used. The following example puts the test method Can_Start_App under the category UI-Basics.

    [Theory, Trait("Category", "UI-Basics")]
[InlineData(false)]
public void Can_Start_App(bool closeAppAfterRun)

    ReSharper's Unit Test Explorer (Extensions > ReSharper > Windows > Unit Tests or Ctrl+Alt+U) shows the method then under category UI-Basics and belonging class:

    InlineData and MemberData

    xUnit allows the definition of multiple input values for a test. Such parametrizable tests needs to have the attribute Theory. Test without parameters are setup as Fact.

    With InlineData the parameters of the text method are filled with the corresponding data of the InlineData definition:

    [Theory]
[InlineData("1.1.1.1", DecoderType.BarcoMonitorWall)]
[InlineData("223.255.255.255", DecoderType.VideoJetDecoder7000)]
public void Can_add_and_configure_decoder_device(string deviceName, DecoderType decoderType)

    With MemberData it is possible to extract the test-data provider for better readability:

    [Theory]
[MemberData(nameof(Data))]
public void Can_switch_Panes(PaneTypeWithPaneName paneTypeWithPaneName)
{
    ...

public static IEnumerable<object[]> Data =>
new List<object[]>
{
    new object[] { new  PaneTypeWithPaneName(...) },
    new object[] { new  PaneTypeWithPaneName(...) },
    ...

    Benefits of Visual Studio Code

    Debugging FAKE scripts

    Debugging FAKE scripts without a proper tool set can be a time consuming and annoying task. But VSC can be used as real IDE with debugging capabilities.

    see also Debugging of FAKE 5 build scripts

    1. Download fake-dotnetcore-portable.zip
      (e.g. from https://github.com/fsharp/FAKE/releases)

    2. Extract the archive to the portable apps folder
      (e.g. c:\_apps\Tools\fake-dotnetcore-portable)

    3. Open Visual Studio Code

    4. Click in the sidebar Extensions and install the following extensions:

      • C#
      • Ionide-fsharp
      • Ionide-FAKE
      • Ionide-Paket

    5. Click in the sidebar Debug

    6. expand configurations-combobox and choose Add Configuration…

    7. in the list select dotnet core

    8. paste this configuration block directly inside the square brackets of the configuration key

      {
"name": “Debug Fake Script”,
"type": “coreclr”,
"request": “launch”,
"program": “c:\_apps\Tools\fake-dotnetcore-portable\fake.dll”,
"args": [“run”, “–nocache”, “–fsiargs”, “–debug:portable –optimize-”, “build.fsx”],
"cwd": “${workspaceRoot}”,
"stopAtEntry": false,
"console": “internalConsole”
}

    9. save the launch.json configuration file

    10. set breakpoints where wanted

    11. select in the configurations-combobox the new configuration Debug Fake Script and run this configuration

    Benefits of LINQPad

    LINQPad is a small utility to interactively writing C# code without the need for a heavyweight IDE. It comes in different edition where also the free edition give some benefits. But only the higher versions are really useful.

    With LINQPad it is much more faster to interact with an application which needs to get UI Automation driven. The most important fact is the almost infinite small compilation and startup time.

    Here are the steps to connect to the ConfigClient:

    1. Start the ConfigClient
    2. Start LINQPad
    3. Under Language select C# Program
    4. Press F4 to open dialog Query Properties where you can adjust some properties
    5. On tab Additional References click button Add NuGet...
    6. In new opened dialog LINQPad NuGet Manager search for FlaUI.UIA3 and add the package with button Add to Query and close the dialog with button Close. this should also add the package _FlaUI Core. If not, add the package like the first package.
    7. On tab Additional Namespace Imports add some default namespaces, like FlaUI.UIA3, FlaUI.Core, FlaUI.Core.AutomationElements and close with OK
    8. Use the script below as a example to interact with the ConfigClient
    void Main()
{
  var processName = "ConfigClient";
  var process = Process.GetProcesses().FirstOrDefault(p => p.ProcessName.Equals(processName));
  var app = Application.Attach(process);

  using (var automation = new UIA3Automation())
  {
    var window = app.GetMainWindow(automation);

    var toolbar = window.FindFirstDescendant("MainMenu : Toolbar");
    var children = toolbar.FindAllChildren();
    foreach (var element in children)
    {
        Console.WriteLine("element: " + element.Name);
        if (element.Name == "System")
        {
            element.Click();
        }
    }
  }
}

    Project documentation

    The project documentation for the BVMS UI Automation project is provided in a modern and state of the art model. It is based on these sources and perspectives:

    • XML Comments
      this is the in-source XML-based code documentation with triple slash (/// <summary>...</summary>).
    • Wiki
      which is hosted on Azure DevOps and based on MarkDown as the markup language and Git as the storage to interact with.
      It is located here: Wiki for BVMS UI Automation

    To keep the documentation parts as most as possible together we use the technique of Git sub-modules. Git sub-modules allow a Git ("main") repository to reference other ("sub") repositories. This gives the possibility to divide a project into several containers (e.g. because of different life cycle or usage needs) but bundle them together at a top level.

    In our concrete case we had to include two Git sub-modules

    • BVMS UI Automation project Wiki
      this Wiki is a Git repository itself containing the MarkDown wiki pages
    • DocFX based Online documentation
      GitHub Pages allow easily to host (static) web sites. These web sites have to be provided in a (dedicated) Git repository. So the output of DocFX (the static web sites) are committed to this Git repository and then pushed to the remote (github.io).

    Advice: To clone a Git repository together with its sub-modules use the option --recursive

    Hints and Marker in the Source Code

    Often enough the developer is in hurry or in a spike mode where he wrote some unclean code lines which he later likes to adjust according the guidelines. Usually later he forget these necessary refactoring. To keep these construction zones on the desk it is wise to add some marker.

    Visual Studio offers such marker management with the Task List. Out of the box these markers are available: TODO, HACK, UNDONE and UnresolvedMergeConflict.

    With ReSharper we have another option, which allows us a better management of such markers and later an easier search.

    So we rely on the enhanced features of ReSharper for our marker management in the project.

    To have an access to our project specific markers it is necessary to add some additional markers.

    1. Open the ReSharper Settings
      Extensions > ReSharper > Options...

    2. Navigate to the setting for the To-do Explorer
      Tools > To-do Explorer

    3. Add the following additional items:

      Name Pattern Icon Selections
      Refactor (?<=\W|^)(?<TAG>REFACTOR)(\W|$)(.*) Warning Match in comments
      Hack (?<=\W|^)(?<TAG>HACK)(\W|$)(.*) Warning Match in comments

    4. Now you can use these new markers, e.g.

      // Refactor low: Method call without using the result?

      You should also add a hint how important that refactoring is. In this case it is low.
      Other priorities should be normal and high.

    5. To see the list of all markers open the window To-do Explorer
      Extensions > ReSharper > Tools > To-do Explorer
